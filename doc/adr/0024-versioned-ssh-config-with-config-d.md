# 24. Versioned SSH Config with config.d/

Date: 2026-03-15

## Status

Accepted

## Context

SSH configuration (`~/.ssh/config`) was unversioned and managed manually. This created friction when:

- Setting up new machines (settings had to be recreated from memory)
- Diagnosing SSH issues (no history of when/why settings changed)
- Managing per-host TERM overrides (discovered when `xterm-ghostty` is not in the terminfo database on remote servers, causing `tmux attach` to fail with "missing or unsuitable terminal")

The specific trigger: SSHing from Ghostty on macOS to a remote Linux server and attaching to tmux would fail because the remote didn't have Ghostty's terminfo. The fix is setting `TERM=xterm-direct` (or `xterm-256color`) for that host, which needs to live somewhere versioned.

The `~/.ssh/config` file also mixes portable settings (1Password IdentityAgent, TERM defaults) with machine-specific entries (LAN IPs, locally-generated includes like colima), making it awkward to version the whole file.

## Decision

Manage SSH config through a `ssh/config.d/` directory in dotfiles, using SSH's native `Include` directive, mirroring the pattern used for `gitconfig.d/`.

### Architecture

```
dotfiles/
└── ssh/
    └── config.d/
        ├── .gitignore   # ignores: hosts, colima
        ├── auth         # Host * - 1Password IdentityAgent
        └── term         # Host * - default TERM=xterm-256color

~/.ssh/
├── config               # Include ~/.ssh/config.d/* + machine-local entries
└── config.d/
    ├── auth -> dotfiles/ssh/config.d/auth   (symlink)
    ├── term -> dotfiles/ssh/config.d/term   (symlink)
    ├── colima           # generated by sshconfig.sh if colima installed
    └── hosts            # machine-local per-host overrides (gitignored)
```

`sshconfig.sh` handles:

1. Creating `~/.ssh/config.d/` with correct permissions
2. Symlinking versioned fragments from `ssh/config.d/`
3. Generating the `colima` fragment if colima is installed
4. Ensuring `~/.ssh/config` starts with `Include ~/.ssh/config.d/*`

### TERM Strategy

- Global default: `xterm-256color` (universally available) in `term` fragment
- Per-host overrides: added to `~/.ssh/config.d/hosts` for servers confirmed to support better terminal types (e.g., `xterm-direct` where available)
- To onboard a new server: check terminfo with `toe -a | grep xterm-direct`, then add a `Host` block to `~/.ssh/config.d/hosts`

### Fragment Ordering

SSH processes `Include` files alphabetically and uses the first match per setting. Specific `Host` blocks in `hosts` (alphabetically later than `auth`/`term`) must appear before `Host *` defaults to take precedence -- this works because SSH evaluates all files and uses the first match, and specific hostnames match before wildcards regardless of file order.

### Alternatives Considered

1. **Build script (like claudeconfig.sh)**: Assemble fragments into a single `~/.ssh/config`. Rejected: SSH's native `Include` handles this without a build step.

2. **Version the entire `~/.ssh/config`**: Simple but mixes portable and machine-specific content. Rejected: colima generates its own SSH config fragment that varies per machine.

3. **Do nothing / manual `~/.ssh/config`**: Current state. Rejected: no history, tedious to recreate on new machines.

## Consequences

### Positive

- Portable SSH settings are versioned and automatically applied on new machines via `install.sh`
- Machine-specific overrides (`hosts`, `colima`) are cleanly separated and gitignored
- Adding per-host TERM overrides has a clear home (`~/.ssh/config.d/hosts`)
- Pattern is consistent with how `gitconfig.d/` works

### Negative

- `~/.ssh/config` itself remains unversioned (just a thin Include + local entries)
- Per-host overrides in `hosts` are not versioned, so they need to be manually recreated or documented when setting up a new machine
